# 端侧推理 · 对外口径与取证

> 赛项：Arm × 通义「手机上的创意 AI 挑战赛」。初赛测试机 = iPhone 15（A16）。
>
> **一条铁律：只说能被「引擎自身日志 + 飞行模式」当场验证的话。**
> A16 无 SME2 硬件，物理上跑不了 SME2，**绝不能声称"SME2 在手机上运行"**——你自己的 MNN 日志会打 `sme2:0`，当场穿帮。

---

## 取证方式（option A：一段 Mac 投屏录屏，同框互证）

USB 把 iPhone（**开飞行模式**）投屏到 Mac（QuickTime「新建影片录制」选 iPhone）→ Xcode 控制台并排 → `Cmd+Shift+5` 录一次桌面。一段视频里同时有：

- **手机侧**：DevHud 实时 `tok/s` + 状态栏 ✈️ 飞行模式 + 思维链可见
- **Mac 侧（引擎自报，最硬证据）**：
  - `[ScholarLLM] load done in 1.83s, use_mmap=true, config=…` ← 模型从本机文件加载
  - `[ScholarLLM] chat stats … decode_speed=17.4 tok/s` ← 实测解码速度，**与 HUD 对上**
  - `The device supports: i8mm:1, sme2:0` / `KleidiAI is running! AccelType is …` ← MNN 自报 ISA

> 飞行模式只关无线电，**USB 投屏/日志照常**。HUD 的 tok/s 会比引擎日志略低（多了 JS 桥接开销）——**略有差异且同步联动，反而比分毫不差更可信**（写死的数字不会跟引擎日志一起跳）。

## 1. 硬指标：本地 CPU · 离线推理（必须站稳）

> 模型 **Qwen3-0.6B** 经 **MNN 端侧引擎**在手机**本地 CPU** 推理，**全程离线、不联网、不走 NPU**。

证据：上面那段录屏（飞行模式 ✈️ + 引擎加载/解码日志）。

## 2. 推理输入输出（考察点）

> 用户输入 → 模型**思维链（可见）** → 最终答案，全程端侧。思维链是 Qwen3 在本机的真实推理过程，非预设文案。

- app 内：聊天气泡顶部「已思考（用时 X 秒）」+ 左竖线灰字推理过程 + 黑字答案（`ReasoningBubble`）。
- 配合 DevHud 的 `tok/s` / 首 token 延迟，呈现"真在本机算"。

## 3. SME2（加分项 · 诚实版）

> 端侧 MNN 已**完整集成 SME2 加速路径**（Arm KleidiAI + MNN SME2 kernel；二进制中 183 个已定义 SME2 符号），并实现运行时能力自适应（`hw.optional.arm.FEAT_SME2` 自动检测）：**在支持 SME2 的 Arm 设备上零改动自动启用**；测试机 iPhone 15（A16）无 SME2，自动回退本地 NEON/i8mm 推理。

- 证据：`nm MNN.framework/MNN | grep -c sme2` → 305；MNN 日志诚实显示 `sme2:0`。
- ❌ 不要说："SME2 正在 iPhone 15 上加速"。
- ✅ 复赛若赛方提供 SME2 设备 → 引擎日志会打 `sme2:1 / KleidiAI is running! AccelType is sme2`，届时直接录真·SME2。
- 🔼 升级（在 A16 上确认日志 `i8mm:1` 后可用）：
  > "在 A16 上走 **i8mm 矩阵加速**（Armv8.6 矩阵 ISA），并为 SME2 前向兼容——下一代 Arm 设备零改动提速。"
